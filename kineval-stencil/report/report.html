Project Pathplan Report

Depth-first search


We see here that the path generated using depth-first search is not optimal. This happens because in depth-first search the nodes are explored only in one direction without any consideration of the goal or the start.

Once it encounters a dead end it has to come all the way back until it finds a neighbor of a node that has been not visited, hence it takes a lot of time to execute. Depth-first search almost never gives the shortest path but it is a easy algorithm to implement when we have no other information.

Breadth-first search

In this search algorithm we explore all the neighbours of a node before going to the next node. Using breadth-first search we will always find the shortest path but as you can see below the number of iterations and the nodes visited is really large, hence the computation time is really high.


Greedy-best first search

As you noticed in DFS and BFS there is no metric as to where the goal is and the direction you are going. So in this algorithm we give it a sense of direction. So the nodes are given priority if they are closer to the goal.


But the only problem with this algorithm is that it maynot always give the shortest path from start to stop. As you can see in Case-3 and Case-4 even though there is an obstacle in the direction towards the goal the algorithm ensures that the nodes closer to the goal are explored first and then the rest. Even though the path given using this algorithm may not be optimal it requires less no of iterations compared to others.

A-star search

In order to avoid the problems caused due to Greedy-Best-First search we added another metric to this algorithm which represents the distance from the current node to the start. This ensures that we don't go to far from the start to avoid the obstacles. As you can see in Case-3 and Case-4, for the same start and goal configurations A-star gave a shorter path compared to Greedy search.
